# PID 제어 튜닝 과정

## 1. 배경 및 목적

명세서 6번 요구사항: **온도 범위 18~27°C 유지율 95% 이상**

PID가 CRAH 공급 온도(`supply_temp_setpoint_c`)를 조절해 서버실 온도를 목표치(24°C)에 유지한다.
PID는 Rule-based 베이스라인 컨트롤러로, RL/MPC 컨트롤러와의 비교 기준이 된다.

---

## 2. 자체 수치 시뮬레이터 구현

PID 튜닝을 위해 Sinergym 없이 서버실 온도를 시간에 따라 적분하는 ODE 기반 시뮬레이터를 구현했다.
(Sinergym과의 오차 검증은 모델 범위 차이로 불가하다고 팀 합의 완료)

### 핵심 수식

```
dT/dt = (Q_in - Q_out) / C_eff
T_zone[t+1] = T_zone[t] + dT/dt × Δt
```

- `Q_in` = IT 발열량 (kW)
- `Q_out` = `min(Q_out_air, Q_out_chiller)` (kW)
- `C_eff` = 서버실 유효 열용량 (kJ/K)

### 유효 열용량 (`C_eff`) 계산 (CPU 400대 + GPU 20대, 중규모 IDC 1홀)

| 구성 요소 | 계산 | 값 |
|-----------|------|----|
| 공기 | 1.2 kg/m³ × 1,500 m³ × 1.005 kJ/kg·K | 1,809 kJ/K |
| CPU 서버 | 400대 × 20 kg × 0.5 kJ/kg·K | 4,000 kJ/K |
| GPU 서버 | 20대 × 60 kg × 0.5 kJ/kg·K (A100×4, 약 60kg) | 600 kJ/K |
| 랙 | 12개 × 100 kg × 0.5 kJ/kg·K (CPU 10 + GPU 2) | 600 kJ/K |
| 구조물 | (바닥·벽 추정) | 2,000 kJ/K |
| **합계** | | **9,009 kJ/K** |

## 3. 설비 설계 반영

### 3-1. CRAH: N+1 이중화

| 항목 | 값 |
|------|----|
| 설치 총 대수 | 4대 (예비 1대 포함) |
| 정상 운영 대수 | 3대 |
| 전 대수 가동 시 총 ṁ | 44.0 kg/s |
| 정상 운영 ṁ (3대) | 33.0 kg/s |
| CRAH 1대당 냉각 용량 | 100 kW (열), ΔT=9°C 기준 |

CRAH 대수가 줄면 ṁ(공기 유량)이 비례해서 함께 감소한다.

### 3-2. 칠러: 2N 이중화 (CRAH와 독립)

| 항목 | 값 |
|------|----|

| COP | 외기 22°C 기준 5.3 |

칠러 용량은 CRAH 대수와 무관하게 별도로 스케일링된다.

## 4. PID 구조 선택: 위치형 → Incremental(속도형)

### 위치형 PID (초기 시도, 폐기)

```
output = Kp·e + Ki·∫e·dt + Kd·de/dt
```

output이 공급 온도 **절댓값**을 직접 계산한다.
초기 오차가 크면 output이 즉시 clamp에 걸려 Kp 크기에 무관하게 bang-bang 동작이 발생한다.

### Incremental PID (채택)

```
supply[t] = supply[t-1] + Kp·Δe + Ki·e·dt + Kd·(Δe/dt)
```

output이 공급 온도 **변화량**을 계산한다.
초기 오차가 커도 supply가 점진적으로 조절되고,
Kp가 "온도 변화량에 대한 공급 온도 변화율"로 물리적 의미를 가진다.

---

## 5. 고정 Setpoint

초기에는 Q_in 기반 동적 setpoint를 시도했으나 폐기했다.

- **문제**: 위기 발생 시 Q_in 급변 → setpoint 급변 → incremental PID에서 delta_error 순간 폭등 → supply 튐
- **채택**: **서버실 목표 온도 24°C로 고정**. `run_pid_loop`에서 `zone_target_c=24.0`으로 주입.

평형 초기화:
```python
supply_eq = zone_target_c - Q_in[0] / (ṁ·cp)  # ≈ 19.2°C
pid._supply = clamp(supply_eq, output_min, output_max)
pid._prev_error = 0.0  # 평형 상태 → 첫 스텝 Kd 항 = 0
```

---

## 6. Anti-windup

위기 상황에서 supply가 물리적 한계에 도달하면 적분항이 무한정 누적(windup)되는 문제가 발생한다.

→ **조건부 적분(conditional integration)** 채택:
supply가 output_min/max에 도달하면 Ki항 누적 중단, 한계를 벗어나면 재개.

## 7. 게인 튜닝

### 7-1. 시스템 파라미터 (정상 운영 CRAH 3대 기준)

```
ṁ = 44.0 × (3/4) = 33.0 kg/s
ṁ·cp = 33.0 × 1.005 = 33.2 kW/K
τ = C_eff / (ṁ·cp) = 9,009 / 33.2 = 272초
IMC 기준: Kp_imc = 1.0,  Ki_imc = Kp / τ ≈ 0.0037
```

Kp=1.0은 IMC 공식에서 ṁ이 소거되어 결정.

### 7-2. Ki sweep (`Kp=1.0, Kd=0.0` 고정, 3개 시나리오 위기 유지율 기준)

| Ki | S1위기% | S2위기% | S3위기% | min% |
|----|---------|---------|---------|------|
| 0.001 | 100.0% | 100.0% | 100.0% | 100.0% |
| 0.002 | 100.0% | 100.0% | 100.0% | 100.0% |
| ... | ... | ... | ... | ... |

→ **Ki=0.001** 채택. IMC 기준값(0.0035)보다 낮게 설정해 안정성 우선.

### 7-3. Kd sweep (`Kp=1.0, Ki=0.001` 고정, S3 복구 수렴 기준)

supply_min=16°C로 확장된 이후, S3(칠러 고장) 복구 구간에서:
- 음수 Kd가 크면(-2 이하): 복구 후 T_zone 과냉각(언더슛) 발생
- 양수 Kd: 복구 중 과냉각 방지, setpoint에 더 근접하게 수렴

```
Kd=-1.0 → S3 복구 후 T_zone 23.68°C (언더슛)
Kd= 0.0 → S3 복구 후 T_zone 23.88°C
Kd= 0.5 → S3 복구 후 T_zone 23.95°C  ← 채택
Kd= 1.0 → S3 복구 후 T_zone 23.95°C
```

→ **Kd=0.5** 채택. S3 복구 후 setpoint 24°C에 가장 근접.

| 시나리오 | 내용 | PID 유지율 |
|----------|------|-----------|

| S3: 칠러 1대 고장 | 전환 160s | **100.0%** |

### 물리적 한계 분석

| 시나리오 | Q_in | Q_out_max | 비고 |
|----------|------|-----------|------|
| S1 (위기) | 187 kW | ~300 kW (열) | Q_out > Q_in, 냉각 여유 있음 |
| S2 (위기) | 144 kW | ~187 kW (COP 3.3) | 외기 46.7°C 이상에서 한계 도달 |
| S3 (전환 중) | 144 kW | 0 kW | 160s 동안 T_zone 상승, 예비 투입 후 복구 |

명세서 기준 기기 스펙상 설계 여유가 충분해 정상 운영 및 위기 전환 후 복구 구간에서 100%를 달성한다.
유지율 100%는 **설계 여유를 올바르게 반영한 결과**이며, PID의 역할은 이를 안정적으로 유지하는 것이다.
